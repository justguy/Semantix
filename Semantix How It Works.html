How Semantix works: from language to governed execution

The control surface is only one layer of Semantix. Underneath it sits a language for fenced semantic computation, a compiler and control plane that turn semantic intent into versioned execution contracts, and a runtime adapter layer that executes approved work under policy, freshness, and audit controls.

Swimlanes

Author -> compile -> review -> execute

The language/runtime thesis becomes real only when the four lanes connect: authoring, compilation, review, and runtime execution all participate in one governed flow.

Lane 1

Authoring

The author writes a Semantix program, not a giant prompt. Semantic regions are fenced and typed from the start.

Example Semantix region

with context { repo_files, auth_routes } {
  constraint VerifyFlow {
    token_must_be_single_use: true,
    must_not_use_prod_smtp: true,
    login_blocked_until_verified: true
  }

  patch ~> "Add email verification to signup flow" :: VerifyFlow;
}

The language primitives carry real semantics

context

Defines the exact working set visible to the semantic step. It is a scope boundary, not a comment.

constraint

Declares the rules a result must satisfy. These compile into validators, policies, or escalation duties.

~>

Marks the semantic boundary explicitly: deterministic code stops here, candidate generation begins here.

Typed admission

Generated output does not become trusted state until it re-enters through deterministic parsing and validation.

Put bluntly, ~> is where the model is allowed to think. Everything downstream in Semantix exists to govern what happens after that invocation.

Lane 2

Compile

The compiler and control plane turn author intent into a versioned execution artifact the system can review and enforce.

Compile-time outputs

IntentContract

Frozen directive, strict boundaries, success state, and contract version.

ExecutionPlan

Versioned graph, dependencies, approval gates, checkpoints, and state-effect previews.

Validators + policy checks

Hard checks, semantic verification plans, and approval-required boundaries are attached before execution starts.

Runtime bindings

Executable nodes are annotated with the runtime family and adapter target that will carry them out later.

What ~> compiles into

with context { routes, auth_logic } {
  constraint VerifyFlow {
    token_must_be_single_use: true,
    must_not_use_prod_smtp: true
  }

  patch ~> "Add email verification endpoint" :: VerifyFlow;
}

ExecutionNode {
  nodeType: "semantic",
  title: "Draft verify endpoint",
  inputSummary: "routes, auth_logic",
  constraints: VerifyFlow,
  approvalRequired: true
}

Every semantic node in the graph is the compiled form of one ~> invocation. The UI does not show ~> directly. It shows the node, critique, diff, and approval machinery produced around it.

Constraint classification and runtime ownership

hard soft_verified hybrid

The compiler classifies each constraint as hard, soft_verified, or hybrid. When the boundary is ambiguous, the safe default is hybrid: run the best deterministic check available, then add semantic verification and escalation rules.

Compile-time behavior

Build the plan, bind capabilities, attach validators, compute approval gates, and generate artifact hashes.

Runtime behavior

Freshness-check mutating actions, reject stale approvals, pause at gates, dispatch approved work, and merge normalized events back in.

What the UI is actually showing when you see a semantic node

SEM node in graph

One compiled ~> invocation rendered as an ExecutionNode.

Context scope

The compiled form of context { ... }: what that semantic step is allowed to see.

Constraints

The compiled form of constraint { ... }, including hard, soft_verified, and hybrid checks.

Weak grounding

A signal that the ~> invocation is bridging assumptions rather than staying well-evidenced.

Diff / proposed changes

The candidate externally visible output produced by the ~> step and later nodes around it.

Approval

The admission decision on whether the result of that ~> invocation may continue toward reality.

The key distinction is that the control surface is showing the consequences of ~> , not the operator itself.

What the reviewer is actually deciding

Did the system understand the task?

Is the intent contract right, or did semantic drift begin before execution?

Is the context bounded correctly?

Are the right sources visible, and are dangerous sources excluded?

Are constraints and policies sufficient?

Do the compiled checks match the real safety requirement, or is review still needed?

What will become real if this continues?

Files, APIs, databases, and external actions are previewed before they are allowed to execute.

Execute

The control plane governs execution. The adapter bridges into a runtime. The runtime actually does the work.

Node lifecycle

How one semantic node becomes governed execution

This is the shortest answer to "how does a node get compiled and executed?" The adapter fits after review and before the runtime backend, not in place of the control plane.

1. Author

A Semantix region declares a semantic task with explicit context, constraints, and a ~> invocation.

2. Classify

The compiler classifies constraints as hard, soft_verified, or hybrid and binds capabilities.

3. Emit

The control plane emits one semantic ExecutionNode for that ~> invocation, plus validators, approval gates, and effect previews.

4. Review

The Review Surface renders graph, inspector, and diff so a human can review the compiled consequences of that ~> call.

5. Dispatch

The runtime registry selects the node's adapter and the control plane dispatches only after approval.

6. Merge back

The adapter returns normalized events, state effects, and risk signals, which the control plane merges into the artifact and audit trail.

Why this page exists

What this companion artifact clarifies

The control surface mock proves the trust model. This page explains the implementation model and language/runtime thesis that sit underneath it.

What actually runs underneath?

The Semantix Control Plane governs artifacts, approval, freshness, audit, and adapter dispatch. Runtime backends perform approved work through an adapter boundary.

Where do adapters fit?

Adapters live below the control plane and above the runtime backend. They translate Semantix nodes into runtime-specific execution and normalize results back up.

Where is the language thesis?

It starts at authoring: context, constraint, ~>, typed admission, and explicit constraint classes all exist before the UI ever renders.

Where is ~> in the product?

Not in the diff panel or graph chrome itself. It lives at the compilation boundary, and every semantic node is the compiled form of one ~> invocation.

Why is this more than a runtime shell?

Because Semantix supplies the language surface, compiler, control-plane contracts, review model, and approval semantics that remain stable even when the runtime changes.

Phase 1 remains intentionally narrow: one adapter, one active backend per run, Codex as the initial execution plug. The architecture stays broader than that first implementation slice.
