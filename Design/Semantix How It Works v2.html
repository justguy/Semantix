Semantix · How it works

A language, a compiler, and a control plane for governed LLM execution.

Semantix compiles human intent into bounded execution contracts. The model proposes inside the contract; deterministic machinery decides what becomes real. Everything else on this page follows from that one shape.

You write
intent
Goal, boundaries, success state. In Semantix source — not in a prompt.
Compiler emits
~>
The semantic boundary. Exactly where the model is allowed to propose.
Runtime produces
ExecutionNode
One reviewable unit: context, constraints, critique, predicted effect.
Human approves
StateEffect
A versioned, freshness-bound change. Only then does reality move.

01 · The full lifecycle in one pageFollow one node

Pick the node: Draft an email-verification endpoint. Walk its five lives — from a line of Semantix source to an audited state change.

What the author writes
1// services/auth/verify.stx 2with context { routes, auth_logic } { 3 4 constraint VerifyFlow { 5 token_must_be_single_use: true, 6 must_not_use_prod_smtp: true, 7 login_blocked_until_verified: true 8 } 9 10 patch ~> "Draft email-verification endpoint" :: VerifyFlow; 11}
What the three primitives do

context scopes the model's working set. Only routes and auth_logic will be visible to this step. Not the whole repo, not billing code.

constraint declares rules the result must satisfy. The compiler will sort these into hard (deterministically checkable), soft (semantically verified), and hybrid.

~> is the only place in the program where a model is allowed to propose. Deterministic code stops at this line. Candidate generation begins at this line.

The rest of Semantix exists to govern what happens after that operator fires.

What the compiler emits
ExecutionNode { id: "n4", nodeType: "semantic", // from ~> title: "Draft verify endpoint", inputSummary: ["routes", "auth_logic"], // from context constraints: { hard: ["token_must_be_single_use"], soft: ["match_existing_middleware_style"], hybrid: ["must_not_use_prod_smtp"] // substring + semantic }, approvalRequired: true, gatingOwner: "semantix", contributingSystems: ["phalanx", "ct_mcp"] }
The translation, line by line

Three source lines become eleven fields. This is the whole compiler thesis: vague instructions turn into enforceable machinery.

~> → nodeType: "semantic". The graph marks this node probabilistic; the runtime knows to wrap it in validation.

context { ... } → inputSummary. The runtime can now enforce that the model sees exactly these two sources — nothing else leaks in.

constraint VerifyFlow → a classified constraints object. must_not_use_prod_smtp is flagged hybrid: a substring check will run first, then a semantic verifier for phrases like "send via production transport."

Ambiguous rules don't get a free pass. They escalate to approvalRequired: true.

The reviewer is approving structure, not text

The node is spotlight in the graph because the reviewer clicked it. The inspector opens only the sections with something to say — no placeholder fields.

CT-MCP has flagged a bridged assumption: the draft calls signToken(), but no such utility exists in the loaded context. The reviewer now has three surgical options, each a plan mutation, not a restart:

Add lib/crypto/tokens.ts to context
Tighten the constraint to forbid inventing utilities
Escalate — mark the node approval-required regardless

Every intervention invalidates exactly n4 and its descendants. The rest of the graph stays valid. The approval clock resets on what changed.

The adapter is the boundary

The control plane never talks to the runtime directly. It asks the registry which adapter owns this node, then hands off through a single interface.

The adapter translates the ExecutionNode into whatever the runtime expects — a prompt, a tool call, a CLI invocation — and normalizes events back into the control plane's vocabulary (StateEffect, RiskSignal, etc).

Every action re-enters through typed gates. Hard checks run first; they can block. Hybrid checks run both substring and semantic passes. Hoplon verifies mutations stay in-scope at the AST level.

Phase 1 uses a single Codex CLI adapter. The architecture is open to more: Phalanx for engineering, provider-backed executors for hosted models, domain runtimes for support or compliance.

Why the loop closes cleanly

The StateEffect is the first thing in this lifecycle with real-world consequences. Everything before it was a proposal.

Three things land together, atomically:

The effect itself — a patch to routes/auth.ts
Enforcement evidence — AST scope pass, policy pass, reversibility class
Audit record — who approved what version of which plan, at what time

Nothing in this record is derived from a chat transcript. The reviewer's approval is bound to the exact artifactHash they saw. If the plan changed between review and execute, the backend rejects the action and asks for re-review.

That's the loop: intent → ~> → reviewed proposal → adapter-dispatched candidate → validated effect → durable record. Drift cannot harden into authoritative state.

03 · What stays invariantWhen the runtime changes, the contract doesn't.

Semantix's value is the left column. The right column can be swapped, upgraded, or multiplied — the approval surface, audit record, and intent contract keep their shape.

Semantix owns stable

The governance contract.
  • The language surface and semantic boundary (~>)
  • Intent contracts and constraint classification
  • Execution plan shape, node types, edge semantics
  • Freshness, invalidation, and approval workflow
  • Review surface information architecture
  • Durable audit artifacts (§17)
If you replace the runtime tomorrow, none of these change. The reviewer opens the same surface, approves the same artifact, signs the same audit record.

Runtime owns pluggable

The execution substrate.
  • Model selection, prompt assembly, token streaming
  • Tool binding and invocation
  • Retry loops, sampling settings, recovery
  • File mutation mechanics and diff generation
  • Capability discovery and feature negotiation
  • Provider-specific quirks and telemetry
Phase 1: Codex CLI. Next: Phalanx, hosted-provider executors, domain runtimes. Each must pass the RuntimeAdapterContract — and that's all Semantix asks of it.
The point isn't to make models deterministic. It's to make the program around the model deterministic enough to engineer, debug, test, review, and trust. Semantix · design thesis
