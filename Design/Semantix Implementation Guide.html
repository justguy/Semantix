01 — OrientationThe spec-to-UI mental model

The three specification documents each describe the same artifact from a different altitude. Holding them in your head as one stack is the fastest way to read the code without getting lost:

DocumentAltitudeWhat it governs in the UI
semantix-overview.md Philosophy & architecture The reasons each panel exists — fenced semantic computation, four invariants, silicon speed vs human speed, the "BS layer." Use it when deciding what to build, not what to render.
Semantix Control Plane UI.md Product contract The normative layout, DTOs, acceptance criteria, and freshness rules. This is the file every component must obey. When the overview and this spec conflict, this one wins.
semantix-review-surface-spec.md Surface-level detail Per-panel field lists, interaction flows, the sample "email verification" scenario. Use it as the checklist when wiring an individual panel.

The UI resolves them by treating the ReviewControlContract from the Control Plane spec as the source of truth, and using the other two to fill in rationale and field detail. Every panel in the prototype is a visual realization of one DTO.

Core principle

Humans approve structure, not vibes. Every design decision — what to surface, what to hide, when to slow down — follows from that line. If a component is making the reviewer approve a reassuring paragraph instead of a compiled artifact, it's wrong.

02 — Code tourFile-by-file map

The project splits the surface into small Babel-loaded JSX files so you can edit one panel without rebuilding the world. Every file below exports its component onto window (required because each <script type="text/babel"> gets its own scope).

scenarios.jsx Three canned runs — SWE, Support, Compliance — shaped exactly like the ExecutionPlan DTO. Swap these for real backend payloads.
app.jsx Advanced three-panel layout: intent bar · graph · inspector · diff · action bar. The canonical Control Surface.
decision.jsx Decision Mode — the one-column "just tell me what to approve" view. Same data, progressive disclosure.
advanced.jsx Wrapper around app.jsx for the chat overlay. Handles the Simple/Advanced toggle cross-talk.
graph.jsx SVG execution graph. Layout, edges, risk coloring, hover/select.
inspector.jsx Per-node inspector: the 10 sections from Control Plane §10.2, conditionally rendered.
diff.jsx State Diff panel. Files, API calls, external actions with policy + reversibility badges.
phases.jsx Silicon-speed / human-speed timeline used in the canvas variants.
ui.jsx Shared primitives: Pill, RiskDot, SectionHeader, DiffLine.
cli.jsx CLI artboard — how semantix run emits the same DTOs rendered above.
Semantix Chat.html Conversational shell. Prompt → review overlay → collapses back as a summary turn.
Semantix Canvas.html Design canvas variant — Decision Mode, Debug Mode, graph layouts, state view, CLI side-by-side.

03 — Data shapescenarios.jsx ↔ the Semantix DTOs

The scenarios file is not a UI concern — it is the pretend backend. Every property maps 1:1 to an abstraction defined in Control Plane §11. When you replace this file with a live API, keep the field names identical and the UI will keep working.

Scenario → ExecutionPlan

scenario.intent→IntentContract (Control Plane §11.3)
scenario.nodes[]→ExecutionNode[] (§11.4)
scenario.edges[]→ExecutionNode.dependsOn (edge list form)
scenario.diff[]→ProposedChange[] / StateEffect[] (§11.6, §12)
scenario.cli→Provenance hint — the command that would produce this plan

Node fields used across the surface

FieldAppears inNotes
nodeTypeGraph shape, Inspector badgesemantic · deterministic · tool · policy_gate · approval · hybrid. Controls node geometry in graph.jsx.
riskGraph color, Decision Mode stripOne of green yellow orange red. Derived from confidence score + policy state per Control Plane §9.4.
groundingInspector · Section Fgrounded · transformed · bridged · unsupported. The "show me where you're guessing" signal.
critiqueInspector · Section G; Decision Mode risk stripThe CT-MCP hook. Only populated when there's something to say; the UI conditionally renders the section.
approval, sideEffectGraph glyphs, Action Bar countersDrive the "pending approvals" and "blocked actions" tallies.
ownerInspector · Overview; provenance chipsWhich subsystem gates this node — semantix · phalanx · hoplon · ct_mcp · guardrail. Single-valued per §11.4.
Fidelity gap to close for production

The prototype carries risk as a pre-baked color. The spec wants a normalized confidenceScore derived from four signals (provenanceStrength · verifierAgreement · retryStability · changeSafety) and banded per §11.4. When you wire a real runtime, compute the score server-side and let the UI band it — don't let the client invent confidence.

04 — Top barThe Intent Contract, rendered

The top bar is the frozen-goal anchor from overview §"Why This Solves The Fix Older Issues Problem." It's implemented in app.jsx as the <IntentBar> section at the top of the three-panel layout.

Required fields (Control Plane §7.2)

  • primary directive → large headline
  • strict boundaries → bullet list with a ⛔ prefix
  • success state → italic trailing line
  • status, planVersion, freshness badge → chips in the right corner

Actions

edit approve reject view history — approving the intent is a separate review event from approving the graph. The prototype currently stubs these; production needs a second approval endpoint.

Why this matters more than it looks

The Intent Contract is the leash described in the overview: "semantic drift is allowed to exist as an internal failed branch, but it is not allowed to harden into authoritative state." Every downstream panel can reference the contract hash; any edit invalidates the entire graph per §14. Treat this card as the root of the dependency tree, not a header decoration.

05 — Panel AExecution Graph

Implemented in graph.jsx as an SVG canvas with pre-computed node positions. Production should swap the hard-coded x,y for an automatic layout engine (React Flow with dagre or elkjs is the spec's suggestion, §18).

Node geometry

  • Rounded rectangle → semantic / deterministic / hybrid work nodes
  • Diamond → policy gates and validation gates
  • Shield outline → approval gates
  • Cylinder / tool glyph → tool nodes (external calls)

Edge semantics (Review Surface §9.6)

Edges currently render as a single style. The spec distinguishes five types; before shipping, encode them as stroke patterns:

  • Solid — execution order
  • Dashed — data dependency
  • Dotted — approval dependency
  • Dashed + muted — fallback path
  • Circular arrow — retry loop

Coloring rule

The four bands from Control Plane §9.4 drive both node fill and the legend at the top-right. When the score is ambiguous, the UI must round down, never up — "risk visibility before convenience" (§4.3). 

// graph.jsx — color resolution, simplified
const risk = node.risk;  // server-provided
const fill = {
  green:  'var(--accent-soft)',
  yellow: 'var(--warn-soft)',
  orange: '#f5e6d4',
  red:    'var(--block-soft)',
}[risk];

Interventions

Five actions per §10.4: edit context edit constraints split node regenerate mark approval-required. All are stubbed in the prototype. In production, each mutates the backend plan and triggers the invalidation set from §14 — never a client-local toggle.

07 — Panel CState Diff Panel

Implemented in diff.jsx. Each ProposedChange renders as a row with four must-show signals per Control Plane §12:

  • Kind — file / api / database / message / external_action icon
  • Policy state — pass review_required block
  • Reversibility — reversible reversible_within_window irreversible. First-class badge, never buried.
  • Hoplon enforcement — AST/scope check result
Hard rule from §12.6

If policyState === 'block', the surface must not render an approval path for that change. Not a disabled button — no button at all. In the code, this is the early-return at the top of <DiffRow>'s action cluster. Don't let product "soften" it.

Diff preview rendering

For file changes, the prototype shows a simple + / − plaintext block. Production should use Monaco (per §18) with syntax highlighting and a ! gutter for Hoplon-flagged lines — see scenarios.swe.diff[1] where the signToken bridged assumption is flagged at line level.

08 — Decision ModeThe simple view

decision.jsx is the "just tell me what to approve" surface — the answer to the pushback that the full control room is too much for 90% of runs. It compresses the three panels into one column:

  1. Intent (one sentence, directive only)
  2. Proposed changes, each with inline approve / block
  3. Per-card expansion reveals the diff preview and — only if risky — a single risk strip with the critique and suggestion
  4. Execution plan is a drawer behind a "See plan" secondary button

The Simple/Advanced toggle in the chat overlay (Semantix Chat.html) flips between DecisionMode and the full app.jsx layout using the same underlying scenario data. This is important: one data source, two projections.

Design intent

Decision Mode is not a dumbed-down Debug Mode — it's the default. Users escalate to Advanced when a card surfaces real risk or when the overall plan needs scrutiny. The control surface should feel like a release PR, not a debugger.

10 — InvariantsFreshness, invalidation, approvals

These are the three rules the prototype stubs but that a real implementation must enforce server-side. They are the difference between a demo and a governance layer.

Freshness (§5, §13)

Every rendered view is keyed to {planVersion, graphVersion, artifactHash, intentContractVersion}. Every mutating action submits those versions; the backend rejects mismatches with a stale-state error. The UI must surface this as a first-class path — refresh + re-open — not an exception toast.

Invalidation set (§14)

EditInvalidatesPreserves
Intent contractEntire graph, all approvals, all diffsPrior audit history
Context on node NN, descendants, their critiques, affected diff entriesUnrelated sibling branches
Constraint on node NN, validators using those constraints, descendantsUnrelated nodes
Tool/capability bindingN, descendants, capability overlays, prior-capability approvalsUnaffected branches
Verifier/model swapConfidence, critique, approvals for affected nodesDeterministic ancestors

Invalidated nodes must transition to stale; any approval attached to a stale node is void until re-reviewed. In the UI this is a status color change plus an inline banner on the inspector. Never silently auto-re-approve.

Policy-blocked ≠ disabled button

Per §16 acceptance criterion 7 and §12.6: a policy-blocked action must not be approvable through any UI path. The diff row in diff.jsx hides the approve control entirely when policy === 'block'. Don't regress this to a greyed-out button — users learn to double-click greyed controls.

11 — Path forwardFrom prototype to production

The prototype covers the surface. Here's what needs backend contracts and what stays client-side:

Backend work

  • Emit ExecutionPlan DTOs matching the shape in scenarios.jsx. Preserve field names.
  • Compute confidenceScore server-side from the four signals; client bands it per §11.4 thresholds.
  • Persist every review event as the durable record from §17 (artifactId, reviewer, action, reason, timestamp, hash).
  • Implement the invalidation set from §14 as a pure function; expose it to both the runtime and the replay tooling.
  • Version-check every mutating endpoint; return stale-state errors with current metadata so the UI can offer "re-open on latest."

Client work

  • Replace hard-coded x,y in graph.jsx with a layout engine (dagre/elkjs under React Flow).
  • Normalize client state by {nodeId, revision} in a Zustand store.
  • Swap the plaintext diff preview for Monaco with Hoplon gutter markers.
  • Wire real intervention endpoints for the five actions in §10.4; treat each as a plan mutation, not a local toggle.
  • Add WebSocket updates so a long-running plan streams node status changes without full-page re-fetches.

What to leave for v2 (explicitly)

The spec is clear about deferrals. Don't get talked into any of these early:

  • Export UI for audit bundles (recording is v1; export is v2)
  • Multi-user presence / threaded review comments
  • Token-level provenance rendering
  • Real-time execution playback studio
  • Branch comparison / scenario simulation

12 — Acceptance checklistBefore calling it v1

Control Plane §16 + Review Surface §16, inlined so you can tick them off against what's built:

  1. A reviewer can inspect the frozen intent contract, status, and plan version before approving.
  2. The execution graph renders node types, dependencies, risk colors, approval gates, and conditional summary fields — no placeholder dashes.
  3. The Node Inspector renders only populated sections and supports local interventions.
  4. The State Diff panel shows proposed changes, policy state, Hoplon enforcement, and reversibility class.
  5. Every mutating review action is freshness-checked; stale approvals are rejected server-side.
  6. Every approve/reject/edit/execute event persists a durable review artifact record.
  7. A policy-blocked action is never approvable through any UI path.
  8. A reviewer can challenge one node and the system regenerates only the affected subgraph.
The one-line test

When a non-Semantix engineer opens the review surface for an unfamiliar run, within 30 seconds they should be able to say what the system is about to do, where it's guessing, and what needs their approval. If they can't, no amount of additional fidelity fixes it — the information architecture is wrong.
